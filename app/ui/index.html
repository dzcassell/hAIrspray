Installing your SASE re-sign certificate

Seeing CERTIFICATE_VERIFY_FAILED errors in the Monitor tab? Your SASE/NGFW (Cato, Zscaler, Palo Alto Prisma, Netskope, etc.) is decrypting outbound HTTPS and re-signing with its own CA. That CA isn't trusted by the container yet. Here's how to fix it on macOS — typically a four-minute job.

Step 1 — Export the re-sign CA from your fabric

For Cato: open the Cato Management Portal, then navigate to System → Certificate → Export CA. Download the certificate (a .crt, .pem, or .cer file) to any convenient location — e.g. ~/Downloads/. Menu paths vary slightly by portal version and by vendor; if you're on Zscaler or Palo Alto, look for the equivalent "Root CA" or "Re-Sign Certificate" export.

Step 2 — Drop it into hAIrspray's certs/ folder

Two equally good ways:

Finder:

  • Open Finder
  • Press ⌘ ⇧ G (Cmd-Shift-G) to open "Go to Folder"
  • Paste ~/Code/hAIrspray/certs and hit Return
  • Drag your exported certificate file into that folder

Terminal:

mv ~/Downloads/cato-root-ca.crt ~/Code/hAIrspray/certs/

The path ~/Code/hAIrspray assumes you followed the macOS quick-start in the project README. If you cloned the repo somewhere else, use that location's certs/ subdirectory instead. Filename doesn't matter as long as it ends in .crt, .pem, or .cer. You can drop multiple certificates here (e.g. test fabric + prod fabric) — all of them will be trusted.

Step 3 — Rebuild the container

hAIrspray reads the cert at container start, so a rebuild is required. In Terminal: 

cd ~/Code/hAIrspray
docker compose down
docker compose build --no-cache
docker compose up -d

Step 4 — Verify it worked

docker compose logs hairspray | grep tls_

You should see a line with "mode": "custom-bundle" and your filename listed under "extras". Example: 

{"mode": "custom-bundle",
 "extras": ["cato-root-ca.crt"],
 "bundle_path": "/tmp/hairspray-ca-bundle.pem",
 "event": "tls_custom_ca"}

Refresh this page and retry any Prompt & Fire targets that were failing — the CERTIFICATE_VERIFY_FAILED errors should be gone.

Troubleshooting. If the log shows "mode": "bypass" instead of "custom-bundle", the file wasn't picked up. Checklist: file extension is .crt/.pem/.cer; file is in PEM format (starts with -----BEGIN CERTIFICATE-----, not a DER/binary blob); file is readable (chmod 644 certs/*.crt); the bind mount is live — check with docker compose exec hairspray ls /etc/ssl/hairspray-extra-ca/.

On Linux hosts the workflow is identical; the certs/ directory lives wherever you cloned the repo (e.g. /opt/hAIrspray/certs/). Everything else is the same. Full reference: certs/README.md.
